# Magnetar — Architecture Decision Records

Lightweight ADRs for the magnetar workspace. Each file captures **one** binding
decision: context, the choice, the consequences, and the date Florentin signed
it off. ADRs are immutable once `Accepted` — if a decision is reversed, write a
new ADR that supersedes the old one and update the old one's status header.

## Status legend

- `Accepted` — In force.
- `Superseded by ADR-NNNN` — Replaced by a later decision.
- `Deprecated` — Withdrawn, no replacement.
- `Proposed` — Drafted, awaiting sign-off.

## How to add an ADR

1. Pick the next free number (highest existing + 1).
2. Copy [`adr/_template.md`](adr/_template.md) into `adr/NNNN-kebab-case-title.md`.
3. Fill the four sections: Context, Decision, Consequences, Status.
4. Sign off at the bottom with the date you got Florentin's explicit go-ahead.
5. Append a row to the [Index](#index) above in the **same** commit.
6. If the ADR supersedes an older one, also flip the older ADR's status header.

## Why one decision per file?

A chronological decision log is great for "what happened on date X" but bad
for "what's the current rule on TLS?". ADRs invert that: one file per
question, one canonical answer per file, and a stable identifier
(`ADR-0005`) you can cite in code comments and commit messages.

## Proposals — implementation maps below the ADRs

Where an ADR locks the **binding decision** ("we will ship X behind
feature Y"), a **proposal** under [`proposals/`](proposals/) locks the
**implementation map** that turns the decision into a landed PR: wire
delta, sans-io additions, runtime ports, four-layer test plan, e2e plan.

See [`proposals/README.md`](proposals/README.md) for the index. The
proposal set currently covers PIP-460 and PIP-466; the PIP-180 and PIP-33
proposals were folded back into their ADRs (0033, 0034) once the work
landed — see each ADR's "Implementation footprint" section.
